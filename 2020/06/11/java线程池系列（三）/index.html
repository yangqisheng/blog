CompletionService是什么

我们先来看一个《Java concurrent in practice》书上的一个例子。有个HTML页面，里面有文字和图片，浏览器渲染这个HTML页面的时候，需要渲染文字和每张图片。图片需要从远程加载图片数据后才能渲染。为了能并行处理，浏览器将渲染文字封装成一个task，下载图片也可以封装成一个任务。不过每张图片的下载都是独立的，为了尽量并行处理，可以将每张图片的下载封装成一个任务。这些独立的任务可以并发的来处理。下面是一个可能的实现代码：

看起来，这个方案，已经尽可能的并行化了。然而，我们看line 24，有可能会出现这种情况：第一张图片很大还在下载中，而第2,3,4张图片都已经下载完成了。但是代码还是卡在line 24，为什么？因为future.get是阻塞的，它需要等到第1个future完成。之所以会出现这种情况，是因为外部没法知道一个Executor里面执行的多个任务哪些已经执行完成了，只能通过future.get来阻塞等待，或者通过非阻塞接口轮训判断。两个方法都不高效。容易想到的一个方案是：某个东西将Executor中完成的future放入一个队列中，外部从这个队列来获取已完成的任务。如果这个队列是空，说明没有一个任务完成，那么外部get时会阻塞，否则就直接能获取到已完成任务。额，这个东西就是CompletionService。看下面类图

CompletionService接口就三个方法：

  1. submit提交任务方法
  2. take获取已完成任务，阻塞式
  3. poll获取已完成，非阻塞式，如果没有任务完成，返回null。

从接口方法可知：CompletionService是一个这样的东西，你可以向他提交任务，然后通过task或poll获取已完成的任务。它和ExecutorService接口一个重要的区别是：CompletionService没有execute方法，即CompletionService本身不能执行任务。ExecutorCompletionService是CompletionService的一个实现，内部注入了一个Executor用于执行任务，还有个BlockingQueue用来保存已完成任务。ExecutorCompletionService是Executor和CompletionService的合体。其内部BlockingQueue是线程安全的。BlockingQueue.take方法是阻塞的，用在这里非常合适。

ExecutorCompletionService的内部实现

先看类图

submit提交任务的时候，将Callable类型的task封装成QueueFuture，以便将completionQueue放进去，当这个QueueFuture执行完成时会调用done方法，而在这个done方法里会将QueueFuture里面Future加入到BlockingQueue，这个时候，如果有外部线程阻塞在take调用，它就会被唤醒，进而拿到future返回。ExecutorCompletionService内部实现代码较简单。

ExecutorCompletionService适合什么场景下使用

适合这样的场景：有一些task，它们可以并发执行，用户只需要拿到第一个完成的task的结果，其它task可以随之取消或忽略。比如生活中的打车聚合平台，当用户选定起始点后，聚合平台就向各个出行公司发送请求，最先成功派车的出行公司获得这个订单，其它的请求就可以取消了。
